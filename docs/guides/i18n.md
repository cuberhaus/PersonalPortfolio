# Internationalization (i18n) Guide

This document describes how translations work in the portfolio after the i18next
migration. All translatable text lives in `locales/{locale}/{namespace}.json`
files. Path-based locale detection (`/`, `/es/…`, `/ca/…`) routes to statically
generated pages — no runtime language switching.

Supported locales: `en` (default), `es`, `ca`.

## Architecture overview

```
locales/
├── en/
│   ├── ui.json              ← shared UI strings (nav, aria, meta, buttons)
│   ├── skills.json          ← translation copy for src/data/skills.json
│   ├── experience.json      ← translation copy for src/data/experience.json
│   ├── education.json       ← …
│   ├── certifications.json
│   ├── demos.json
│   ├── work_projects.json
│   └── <demo-slug>.json     ← per-demo page/component copy
├── es/
│   └── (same structure)
└── ca/
    └── (same structure)

src/data/
├── skills.json              ← identity-only (non-translatable fields)
├── experience.json
├── education.json
├── certifications.json
├── demos.json
└── work_projects.json
```

**i18next** initialises at build time (`src/i18n/i18next.ts`) with all JSON
resources imported statically. No runtime bundle is shipped to the browser.

**inlang** reads the same `locales/` files natively via `@inlang/plugin-i18next`
— no bridge script or export/import step needed.

## Quick chooser

| I want to translate…                                         | Where to put it                     |
| ------------------------------------------------------------ | ----------------------------------- |
| A short shared UI string (button label, nav item, ARIA text) | `locales/{locale}/ui.json`          |
| A structured record (demo card, job, skill, certification)   | `locales/{locale}/{data-file}.json` |
| Per-page / per-component copy with HTML or placeholders      | `locales/{locale}/<demo-slug>.json` |

## Pattern A — UI strings (`ui.json`)

Flat key-value dictionaries consumed through `useTranslations(lang)` which wraps
i18next's `getFixedT(lang)`:

```json
// locales/en/ui.json (excerpt)
{
  "nav.home": "Home",
  "demo.viewSource": "View Source on GitHub"
}
```

Usage in any `.astro` file:

```astro
---
import { getLangFromUrl, useTranslations } from '../i18n/utils';
const lang = getLangFromUrl(Astro.url);
const t = useTranslations(lang);
---

<a href="...">{t('demo.viewSource')}</a>
```

Rules:

- Keys are dotted strings like `nav.*`, `demo.*`, `aria.*`, `meta.*`.
- `t(key)` falls back to `en` if a translation is missing.
- Don't put long prose or HTML here — use a per-demo namespace.

---

## Pattern B — Structured content (data files + locale JSON)

Identity (non-translatable) fields live in `src/data/<file>.json` as a plain
array. Translatable copy lives in `locales/{locale}/<file>.json` indexed by
array position:

```json
// src/data/experience.json (identity-only)
[{ "company": "Deloitte", "link": "https://...", "logo": "/logos/..." }]
```

```json
// locales/en/experience.json (translations)
{
  "0": { "role": "AI & ML Engineer", "period": "Sep 2024 — Present", "bullets": ["..."] }
}
```

Components merge them via `flattenForLocale`:

```astro
---
import { experience, getTranslations } from '../data/loaders';
import { flattenForLocale } from '../i18n/load';

const lang = getLangFromUrl(Astro.url);
const entries = flattenForLocale<ExpEntry>(experience, lang, getTranslations('experience', lang));
---
```

### Demos accessor

Demos get a typed helper so pages never touch the raw JSON:

```astro
---
import { getDemo } from '../../i18n/demo';
const demo = getDemo('jsbach', lang);
---

<h1>{demo.title}</h1>
```

Fields on a `Demo`: `slug`, `title`, `description`, `tags[]`, `icon`,
`github?`, `image`, `lead`, `badge`, `metaDescription`, and optionally
`metaTitle`, `aboutTitle`, `aboutDescription`, `hints`. Falls back to `en` if
missing.

---

## Pattern C — Per-demo namespace JSON

For copy specific to one demo page or client island, especially with inline HTML
or `{0}` placeholders:

```json
// locales/en/joc-eda.json
{
  "uploadBtn": "Upload Replay File",
  "loadedFile": "Loaded \"{0}\" — opening viewer..."
}
```

These are imported as i18next namespaces or directly as JSON in the page
component.

Rules:

- One namespace per demo. Naming: `<slug>.json`.
- Do **not** duplicate card-level fields (title, description, badge, lead) —
  those live in `locales/{locale}/demos.json`.
- Always provide an `en` entry; other locales fall back to `en`.

---

## Locale detection & routing

Source of truth in `src/i18n/utils.ts`:

- `getLangFromUrl(Astro.url)` → `'en' | 'es' | 'ca'`
- `useTranslations(lang)` → `t(key)` backed by i18next `getFixedT`
- `getRouteFromUrl(url)` → route without the locale prefix

Client components receive `lang` as a prop.

## inlang integration

[inlang](https://inlang.com) reads `locales/` directly — no export/import
scripts. The `@inlang/plugin-i18next` plugin matches the `locales/{locale}/{namespace}.json`
path pattern.

### Setup

1. Install the **Sherlock** VS Code extension (`inlang.vs-code-extension`) for
   inline translation previews and editing.
2. Config: `project.inlang/settings.json`.

### Workflow for translators

1. Edit `locales/es/*.json` and/or `locales/ca/*.json` using:
   - **Sherlock** (VS Code) — inline editing with context
   - **Fink** (inlang web editor) — no local setup needed
   - Direct file editing — simple key-value JSON
2. Run `npm test` to confirm parity and schema checks pass.
3. Commit.

### Lint rules

- **Missing translation** — keys in `en` not present in other locales.
- **Empty pattern** — keys with empty string values.

These appear automatically in Sherlock and Fink.

---

## Decision flow for new content

1. **Short reusable UI string?** → add to `locales/{locale}/ui.json` in all
   three locales.
2. **Structured data reused across pages** (card, job, cert)? → add identity to
   `src/data/<file>.json`, add translations to all `locales/{locale}/<file>.json`
   at the next index.
3. **Per-page copy** with HTML or placeholders? → create/extend
   `locales/{locale}/<slug>.json`.

When in doubt, prefer B for content and A for UI chrome.

## Adding a new locale

The locale list is defined once in `src/config/locales.ts` (`LOCALES` array).
All translation loading uses `import.meta.glob` via `src/i18n/locale-glob.ts`,
so no per-locale static imports exist — the glob picks up new directories
automatically.

### Checklist (e.g. adding `fr`)

1. **`src/config/locales.ts`** — add `'fr'` to the `LOCALES` array.
2. **Create locale files** — copy `locales/en/` to `locales/fr/` and translate.
3. **`src/i18n/ui.ts`** — add `fr: 'Français'` to the `languages` map.
4. **`project.inlang/settings.json`** — add `"fr"` to the `locales` array.
5. **(Optional)** Deploy workflow / CV — add the CV filename for the new
   language if applicable.

That's it. Everything else (i18next, data loaders, demo translations, hreflang
links, routing) reads from `LOCALES` and the glob loader automatically.

### How it works

`src/i18n/locale-glob.ts` uses Vite's `import.meta.glob('../../locales/**/*.json', { eager: true })`
to load all locale files at build time. Helper functions (`getI18nextResources`,
`getDataTranslations`, `getDemoTranslations`, `getDemoT`) build the runtime
objects from the glob output, keyed by `LOCALES`.

### Compile-time safety

The `languages` map in `src/i18n/ui.ts` is typed `Record<Locale, string>` —
TypeScript will error until you add the display name for the new locale.
