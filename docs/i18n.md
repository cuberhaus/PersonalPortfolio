# Internationalization (i18n) Guide

This document describes how translations work in the portfolio and when to use each
pattern. User-facing copy should live in JSON whenever it may be edited through
Crowdin; TypeScript may wrap that JSON with types and accessors, but should not be
the source of translatable strings. Path-based locale detection (`/`, `/es/…`,
`/ca/…`) is already uniform everywhere.

Supported locales: `en` (default), `es`, `ca`.

## Quick chooser

| I want to translate…                                               | Use pattern                            |
| ------------------------------------------------------------------ | -------------------------------------- |
| A short shared UI string (button label, nav item, ARIA text)       | **A — `ui.ts`**                        |
| A structured record reused across the site (demo card, job, skill) | **B — localized JSON**                 |
| Per-demo / per-component copy, placeholders, inline HTML snippets  | **B JSON + C typed loader, if needed** |

---

## Pattern A — `src/i18n/ui.ts` (shared UI strings)

Flat dotted-key dictionaries per locale, consumed through a `t()` function.

```ts
// src/i18n/ui.ts (excerpt)
export const ui = {
  en: { 'nav.home': 'Home', 'demo.viewSource': 'View Source on GitHub' },
  es: { 'nav.home': 'Inicio', 'demo.viewSource': 'Ver código en GitHub' },
  ca: { 'nav.home': 'Inici', 'demo.viewSource': 'Veure codi a GitHub' },
} as const;
```

Usage in any `.astro` file:

```astro
---
import { getLangFromUrl, useTranslations } from '../i18n/utils';
const lang = getLangFromUrl(Astro.url);
const t = useTranslations(lang);
---

<a href="...">{t('demo.viewSource')}</a>
```

Rules:

- Keys are dotted strings like `nav.*`, `demo.*`, `aria.*`, `meta.*`.
- `t(key)` falls back to `en` if a translation is missing.
- Don't put long prose or HTML here — use localized JSON, with a typed loader if
  the copy is feature-specific.

## Pattern B — localized JSON resources

Used for any data that is naturally a list of records and lives in `src/data/`:

```
demos.json
experience.json
skills.json
work_projects.json
education.json
certifications.json
```

Each file uses the canonical localized envelope:

```json
[
  {
    "identity": {
      "slug": "example",
      "icon": "microscope"
    },
    "copy": {
      "en": { "title": "Example", "description": "English copy" },
      "es": { "title": "Ejemplo", "description": "Texto en español" },
      "ca": { "title": "Exemple", "description": "Text en català" }
    }
  }
]
```

`identity` holds stable, non-translatable fields: slugs, icon keys, URLs,
backend IDs, image paths, and other values code depends on. `copy` holds all
user-facing strings keyed by locale. Crowdin should edit `copy` values, not
TypeScript dictionaries or identity fields.

The default locale (`en`) is required; other locales may fall back to `en` while
in progress. Shape and identity parity are enforced by
`src/__tests__/content-parity.test.ts` and `data-integrity.test.ts`.

### Demos accessor

Demos get a typed helper so pages never touch the raw JSON:

```astro
---
import { getDemo } from '../../i18n/demo';
import { getLangFromUrl } from '../../i18n/utils';

const lang = getLangFromUrl(Astro.url);
const demo = getDemo('jsbach', lang);
---

<h1>{demo.title}</h1>
<p set:html={demo.lead} />
```

Fields on a `Demo` record: `slug`, `title`, `description`, `tags[]`, `icon`,
`github?`, `image`, `lead`, `badge`, `metaDescription`, and optionally
`metaTitle`, `aboutTitle`, `aboutDescription`, `hints`. `getDemo` falls back to
`en` if the slug is missing in the requested locale.

For other content types (`experience`, `skills`, …) the pages import the three
JSONs directly and pick by `lang`. They follow the same parity rules.

---

### Demo and client-island copy

Demo cards, headers, SEO fields, and about text belong in `src/data/demos.json`.
If a React island needs extra translatable labels, placeholders, notebook copy,
pipeline step labels, or other component-specific prose, put that copy in a JSON
resource under `src/i18n/demos/`, for example:

```json
{
  "copy": {
    "en": { "uploadBtn": "Upload replay file", "loadedFile": "Loaded \"{0}\"" },
    "es": { "uploadBtn": "Subir replay", "loadedFile": "Cargado \"{0}\"" },
    "ca": { "uploadBtn": "Pujar replay", "loadedFile": "Carregat \"{0}\"" }
  }
}
```

Keep placeholders such as `{0}` stable across locales so components can reuse the
same formatting logic.

## Pattern C — typed loaders around JSON

For copy that is specific to one page or client island, TypeScript files may
import JSON and expose a typed accessor. They should not own user-facing strings.

```ts
// src/i18n/demos/joc-eda-page.ts
import copy from './joc-eda-page.json' with { type: 'json' };

export const JOC_EDA_COPY = copy.copy;

export function getJocEdaCopy(lang: string) {
  return JOC_EDA_COPY[(lang in JOC_EDA_COPY ? lang : 'en') as keyof typeof JOC_EDA_COPY];
}
```

Rules:

- One JSON resource per demo/component when extra copy is needed. Naming
  convention: `<slug>-page.json` (page UI) or `<slug>-demo.json`
  (client-island copy). A matching `.ts` loader is allowed only for types,
  validation, and fallbacks.
- Do **not** duplicate card-level fields (title, description, badge, lead,
  about) — those live in `demos.json`.
- Always provide an `en` entry; other locales fall back to `en`.
- Legacy `TRANSLATIONS` objects in `.ts` files predate the Crowdin workflow.
  When touching them, migrate the strings to JSON rather than adding more copy
  to TypeScript.

## Locale detection & routing

One source of truth in [`src/i18n/utils.ts`](../src/i18n/utils.ts):

- `getLangFromUrl(Astro.url)` → `'en' | 'es' | 'ca'` based on the first path
  segment.
- `useTranslations(lang)` → `t(key)` accessor for pattern A.
- `getRouteFromUrl(url)` → route without the locale prefix (for language
  switchers).

Client components receive `lang` as a prop rather than reading the URL
themselves.

## Decision flow for new content

1. Is it a **short reusable UI string**? → add to `ui.ts` under a clear
   namespace (`nav.*`, `demo.*`, `aria.*`, …) in all three locales.
2. Is it **structured data reused across pages** (card, list item, bio entry)?
   → add a row to the three `*.json` triples in `src/data/`. Keep field order
   identical. Run vitest to confirm parity tests pass.
3. Is it **copy for a single page/component** with HTML or placeholders?
   → create / extend JSON in `src/i18n/demos/<slug>-page.json` or
   `<slug>-demo.json`; add a small `.ts` loader only if the component needs a
   typed accessor.

When in doubt, prefer B for content and A for UI chrome. Pattern C is a loader
layer, not a translation store.

## What's intentionally _not_ here

- No runtime i18n library (no `i18next`, `react-intl`, etc.). All dictionaries
  are plain objects / JSON imported at build time.
- No ICU message format. Placeholders use simple `{0}` replacement done
  manually at the call site (see `joc-eda-page.ts`).
- No pluralization helper. If you need one, add it narrowly at the call site.

---

## Why not `i18next` (or similar)?

Short answer: **the cost outweighs the benefit for a static 3-locale Astro
site.** i18next shines when you need things this project doesn't have:

| i18next feature                           | Needed here?                                      |
| ----------------------------------------- | ------------------------------------------------- |
| Runtime language switching without reload | No — `/`, `/es`, `/ca` are separate static routes |
| ICU pluralization (`1 item` / `2 items`)  | No — no dynamic counts in copy                    |
| Gender / context variants                 | No                                                |
| Lazy-loaded translation bundles           | No — all JSON is tiny                             |
| Backend fetch of translations             | No — build-time is fine                           |
| Dates / numbers formatting                | Marginal — `Intl.*` covers it                     |

What you'd gain: a familiar `t('key', { count, name })` API.
What you'd pay:

- **+60–100 KB** shipped to the client if any island uses it.
- **Runtime init** in every Astro island — more moving parts.
- **Tooling overhead** (namespaces, extractors, parser config) for what is
  currently a one-file lookup.
- **Loss of static safety**: today `useTranslations` + `keyof typeof ui` gives
  a **compile-time error** on unknown keys. i18next keys are runtime strings
  unless you layer `i18next-resources-for-ts` on top.
- **Doesn't solve the real pain points** (file-naming in `src/i18n/demos/`,
  pattern-C duplication risk, lack of parity tests for `ui.ts`).

### When it _would_ become worth it

Revisit this choice if any of these become true:

1. Client-side language switching without a full page reload is required.
2. ICU plurals / gender are needed in 20+ places.
3. Locales grow to 10+ and bundle splitting matters.
4. The translation-management workflow starts requiring ICU/i18next namespaces
   beyond plain JSON resources.
5. The site is rewritten as a Next.js / React SPA where `react-i18next` is the
   default.

### Middle-ground options

Before reaching for i18next, consider:

- **[Paraglide JS](https://inlang.com/m/gerre34r/library-inlang-paraglideJs)**
  (`@inlang/paraglide-js`) — compiles translations to tree-shakeable typed
  functions, no runtime library, designed for Astro/SvelteKit.
- **`@nanostores/i18n`** — ~1 KB, works with Astro islands, still build-time.
- **Staying put and polishing** this setup: migrate legacy TS demo dictionaries
  to JSON resources under `src/i18n/demos/`, add a parity test for `ui.ts`, add
  a `getDemoCopy(slug, lang)` helper analogous to `getDemo`.

Given the current size and that tests already pass, the polish path is the
highest-value option.

## Testing

- `src/__tests__/data-integrity.test.ts` — shape/field checks per JSON.
- `src/__tests__/content-parity.test.ts` — en/es/ca must cover the same slugs
  and field sets.
- `src/__tests__/i18n-utils.test.ts` — locale detection and route stripping.

Run all of them with:

```sh
npx vitest run
```
